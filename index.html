System Architecture (SARI)
Section 3.1 — Four-entity cloud-IoT security model
The HyMLCPP model defines four system entities: Data Steward (DSid) encrypts data before upload; Cloud Sphere (CSid) stores, computes, and shares data across 4 cloud types; End User (EUid) decrypts with a key; and Unauthorized Party (UnPid) is the adversary.

⚠️ Issue 1 — Data Leakage

Third-party cloud providers can access data. Multiple entities have access for trade purposes. Even legitimate enterprises may expose data accidentally.

Risk: Unauthorized data disclosure even in encrypted storage
💥 Issue 2 — DDoS Attacks

Distributed Denial of Service overwhelms cloud servers with massive traffic from multiple sources, disrupting legitimate access.

Risk: Service disruption, financial loss, data unavailability
⏱️ Issue 3 — Computational Cost

Full image encryption is extremely slow. In real-time IoT environments, speed vs security tradeoff becomes critical.

Risk: System slowdown, battery drain in IoT devices
🔓 Issue 4 — Access Control

Standard CP-ABE encryption reveals access policies publicly, creating privacy vulnerabilities. Key management remains centralized and risky.

Risk: Policy leakage, single point of failure
🎯 Design Goals of HyMLCPP
DG1: Enable secure data sharing between stewards with proper security & privacy
DG2: Prevent unauthorized access — differentiate fault vs non-fault users
DG3: Perform efficient computation over data for improved preservation
DG4: Maintain confidentiality during data distribution with authorized persons only
🖼️ Cryptographic Technique · SA3

Bit Plane Decomposition

Every pixel in a digital image is stored as an 8-bit number (0–255). Bit Plane Decomposition alag karta hai each bit layer ko. Higher bit planes (4–7) contain more visual information — these are the ones that need encryption!

💡 Key Insight: Instead of encrypting the ENTIRE image (slow!), HyMLCPP uses SVM to identify which bit planes contain important data and encrypts ONLY those. This reduces computational time drastically while maintaining security.

🔍 How This Works in HyMLCPP
  • Image decomposed into 8 bit planes (0 = LSB, 7 = MSB)
  • SVM classifier identifies planes 4–7 as "important" (high visual information)
  • Only those planes get encrypted using chaotic maps + AES-256
  • Planes 0–3 carry minimal visual info — skipping them saves ~50% encryption time
  • During decryption, only the encrypted planes need decryption, then recombined

SVM Decision Boundary

Support Vector Machine (SVM) ek ML model hai jo data ko 2 classes mein divide karta hai using a hyperplane. HyMLCPP mein SVM decide karta hai: "Kya yeh image portion encrypt karna chahiye?"

📐 Formula: Z = V·M + a where Z = hyperplane, V = weight vector, M = input, a = threshold bias
If Z > 0 → Encrypt this portion | If Z ≤ 0 → Skip encryption (save computation)

💡 Why Linear Kernel?
  • Experiments showed linear kernel outperformed RBF & polynomial in both accuracy and speed
  • No non-linear patterns observed in bit-plane feature distribution
  • 5-fold cross-validation confirms generalization across unseen image data
  • Parameter C tuned via grid search to balance margin size vs misclassification

HyMLCPP ka unique feature — sirf detect nahi karta, predict bhi karta hai! Bayesian framework use karke ye model future DDoS attack ki probability calculate karta hai current network indicators se.

📐 Bayesian Formula: P(Attack | Evidence) = P(Evidence | Attack) × P(Attack) / P(Evidence)
Multiple evidence factors combine to give final attack probability estimate.

🔍 Limitations & Future Work

Research Gaps & Limitations

Koi bhi research paper complete nahi hoti bina apni limitations acknowledge kiye. Yeh section honestly dikhata hai kahan HyMLCPP aur improve ho sakta hai.

01
🔄 Chaotic Map Synchronization Challenge
Chaotic systems ki extreme sensitivity to initial conditions ka matlab hai — sender aur receiver ka perfect sync rehna zaroori hai. Channel noise mein desynchronization ho sakta hai. SKI technique partially mitigate karta hai but not fully.
02
⚡ Real-Time IoT Constraints
IoT devices mein limited battery aur processing power hota hai. AES-256 encryption + ML inference ka combined overhead in-device pe computationally heavy ho sakta hai for edge IoT scenarios.
03
🌐 Multi-Cloud Scalability
Experiments primarily performed on simulated cloud environments. Large-scale production deployment across heterogeneous multi-cloud environments (AWS + Azure + GCP) ka performance yet to be validated.
04
🔁 Adversarial ML Attacks
SVM and Random Forest are vulnerable to adversarial examples. Attacker carefully crafted traffic patterns ko use kar ke classifier ko fool kar sakta hai. Adversarial robustness testing not included in this paper.
05
📊 Differential Privacy Epsilon Selection
Optimal ε (epsilon) value selection depends on data sensitivity — but in the paper, epsilon is treated as a fixed hyperparameter. Adaptive/dynamic epsilon based on real-time data sensitivity is future work.
06
🧪 Post-Quantum Cryptography
AES-256 is currently secure, but quantum computers could break it. The paper does not address quantum-resistant alternatives (lattice-based, hash-based cryptography) which will be necessary in the near future.
